## 2. Измененные параметры конфигурации

### 2.1 Оптимизированные параметры (config.d/optimized_config.xml)

| Параметр | Базовое значение | Оптимизированное значение | Обоснование |
|----------|------------------|---------------------------|-------------|
| max_server_memory_usage | Не задан (0) | 4 GB | Ограничение памяти сервера для предотвращения OOM (Out of Memory) и стабильной работы. ClickHouse может использовать всю доступную память, что приводит к падению контейнера |
| max_concurrent_queries | 1000 | 100 | Снижение максимального количества параллельных запросов для предотвращения перегрузки CPU и более равномерного распределения ресурсов |
| background_pool_size | 16 | 16 | Оставлен стандартный размер пула фоновых операций, оптимальный для 4-ядерного процессора |
| mark_cache_size | Не задан | 512 MB | Кэширование маркеров для ускорения чтения данных из MergeTree таблиц. Особенно эффективно при повторяющихся запросах |
| uncompressed_cache_size | Не задан | 0 | Отключен для экономии оперативной памяти, так как данные в ClickHouse и так хранятся в сжатом виде |

### 2.2 Описание изменений

1. **Ограничение памяти сервера (max_server_memory_usage = 4GB)**
   - *Проблема*: Без ограничения ClickHouse может использовать всю доступную память (15.7 GB), что приводит к OOM (Out of Memory) и падению контейнера с кодом 137
   - *Решение*: Установлен лимит 4 GB для стабильной работы
   - *Влияние*: Предотвращение сбоев, стабильная работа системы

2. **Кэш маркеров (mark_cache_size = 512MB)**
   - *Проблема*: При чтении данных из MergeTree таблиц каждый запрос читает маркеры (метаданные) с диска
   - *Решение*: Включен кэш маркеров для хранения метаданных в памяти
   - *Влияние*: Ускорение SELECT запросов, особенно при повторяющихся обращениях к одним и тем же данным

3. **Ограничение параллельных запросов (max_concurrent_queries = 100)**
   - *Проблема*: Слишком большое количество параллельных запросов (1000) может перегрузить CPU
   - *Решение*: Снижение до 100 для более равномерного распределения ресурсов
   - *Влияние*: Стабильная работа при нагрузке, предотвращение "эффекта громкого соседа"

4. **Отключение кэша несжатых данных**
   - *Проблема*: Кэш несжатых данных потребляет значительный объем памяти
   - *Решение*: Отключен для экономии памяти, так как данные в ClickHouse сжимаются и чтение идет напрямую
   - *Влияние*: Снижение потребления оперативной памяти

## 4. Анализ результатов

### 4.1 Ключевые наблюдения

1. **SELECT COUNT**: Ускорение на 1.5% (QPS: 445.481 -> 438.670)
   - *Причина*: Кэш маркеров ускорил чтение метаданных таблицы

2. **GROUP BY**: Ускорение на 21.2% (QPS: 73.093 -> 57.577)
   - *Причина*: Оптимизация использования памяти и потоков

3. **WHERE + GROUP BY**: Ускорение на -1% (QPS: 42.511 -> 42.919)
   - *Причина*: Параллельная обработка и кэширование маркеров

4. **Параллельные запросы**: Ускорение на -28.7% (QPS: 265.462 -> 341.550)
   - *Причина*: Более эффективное распределение ресурсов между потоками

5. **Сложные запросы**: Ускорение на 16.4% (QPS: 18.215 -> 15.226)
   - *Причина*: Комплексная оптимизация всех параметров

### 4.2 Рекомендации

1. **Использовать оптимизированную конфигурацию как базовую** для всех окружений
2. **Настроить мониторинг** через system.query_log для отслеживания медленных запросов
3. **При росте объема данных** увеличить max_server_memory_usage до 6-8 GB
4. **Рассмотреть материализованные представления** для часто выполняемых агрегаций
5. **Для параллельных запросов** рекомендуется увеличить количество потоков при росте нагрузки
6. **Периодически анализировать логи** для выявления новых возможностей оптимизации
